## 1. Metadata API

### What is it?

`metadata` is a **named export constant** available in Next.js App Router (v13+). It lets you define SEO-related HTML `<head>` tags — such as `<title>` and `<meta name="description">` — directly from your page or layout file, without manually writing any HTML.

### What is it used for?

- Setting page titles and descriptions for search engine indexing
- Providing social media preview data (Open Graph, Twitter Card)
- Defining canonical URLs, robots directives, and other meta tags
- Centralising SEO configuration at the layout or page level

### Constraints to remember

| Rule | Detail |
|------|--------|
| **Server Components only** | `metadata` export is not allowed in Client Components (`"use client"`) |
| **No mixing static + dynamic on same route** | A route can export either `metadata` (static) or `generateMetadata()` (dynamic), not both |
| **Layout vs Page** | `metadata` in a layout applies to all child routes; a page-level `metadata` overrides/merges with it |

### How to use it

```typescript
// app/page.tsx  OR  app/layout.tsx
import type { Metadata } from "next";

export const metadata: Metadata = {
  title: "Metadata",
  description: "This is static metadata",
};

export default function Page() {
  return <main>Hello World</main>;
}
```

**What Next.js renders in `<head>`:**
```html
<title>Metadata</title>
<meta name="description" content="This is static metadata" />
```

## 2. Title Metadata

### What is it?

The `title` field inside `Metadata` supports three sub-options beyond a plain string: `default`, `template`, and `absolute`. These allow you to build a **title inheritance system** across nested layouts and pages.

### What is it used for?

| Option | Purpose |
|--------|---------|
| `default` | Fallback title used when a child route does **not** define its own title |
| `template` | A pattern string with `%s` placeholder — child page titles are injected at `%s` |
| `absolute` | Forces a specific title on a page, **ignoring** any `template` defined in a parent layout |

### How to use it

#### Root Layout — define the template

```typescript
// app/layout.tsx
import type { Metadata } from "next";

export const metadata: Metadata = {
  title: {
    default: "Auth | Kiwadev",       // shown when no child sets a title
    template: "%s | Kiwadev",        // child title replaces %s → "Sign In | Kiwadev"
    absolute: "",                    // empty here; would override template if set
  },
  description: "Generated by create next app",
};
```

#### Child Page — inherit the template

```typescript
// app/sign-in/page.tsx
import type { Metadata } from "next";

export const metadata: Metadata = {
  title: "Sign In",
  // Result in <head>: "Sign In | Kiwadev"  (template applied from layout)
};
```

#### Child Page — bypass the template with `absolute`

```typescript
// app/sign-up/page.tsx
import type { Metadata } from "next";

export const metadata: Metadata = {
  title: {
    absolute: "Sign Up Absolute",
    // Result in <head>: "Sign Up Absolute"
    // The layout's template "%s | Kiwadev" is completely ignored
  },
};
```

#### Decision flow

```
Does the page define metadata.title?
├── No  → use layout's `default`
├── Yes (plain string or object without `absolute`)
│       → insert into layout's `template`  e.g. "Sign In | Kiwadev"
└── Yes (object with `absolute`)
        → use it as-is, template is ignored  e.g. "Sign Up Absolute"
```

## 3. Favicon

### What is it?

A favicon is the small icon shown in browser tabs, bookmarks, and history. Next.js uses **file-based convention** to pick it up automatically — no configuration needed.

### What is it used for?

Branding your site in browser tabs. Next.js handles the `<link rel="icon">` tag for you.

### How to use it

Simply place a `favicon.ico` file directly inside the `/app` directory:

```
app/
├── favicon.ico     ← Next.js detects this automatically
├── layout.tsx
└── page.tsx
```

Next.js will inject the following into `<head>` without any code changes:

```html
<link rel="icon" href="/favicon.ico" />
```

**Other supported file names for extended coverage:**

| File | Purpose |
|------|---------|
| `icon.png` / `icon.svg` | General app icon |
| `apple-icon.png` | iOS home screen icon |
| `favicon.ico` | Classic browser tab icon |

## 4. OG Images

### What are OG Images?

Open Graph (OG) images are preview images that appear when a URL is shared on social platforms (Twitter/X, Facebook, LinkedIn, Discord, WhatsApp, etc.). They are defined via `<meta property="og:image">` in the HTML head.

### Why use them?

- **Higher engagement** — posts with images get significantly more clicks
- **Better brand recognition** — consistent visual identity across shares
- **Professional appearance** — makes your links look polished rather than plain text
- **SEO signal** — search engines use OG data for richer results

---

### 4.1 Static OG Images

#### What is it?

Pre-created image files (`.png`, `.jpg`, `.webp`) placed in the file system and referenced statically.

#### How to use it

Place an image named `opengraph-image.png` inside any route folder:

```
app/
├── opengraph-image.png   ← auto-detected by Next.js for the root route
├── blog/
│   └── opengraph-image.png  ← applies to /blog and its children
```

Next.js automatically adds the correct `<meta>` tags — no code needed.

#### Pros & Cons

| Pros | Cons |
|------|------|
| Zero code, instant setup | No personalisation per user or content |
| Fast (no runtime generation) | Must be manually created for every page needing a unique image |
| Works offline / no edge compute | Cannot include dynamic data (post title, user name, etc.) |
| | Storage grows with the number of pages |

---

### 4.2 Dynamic OG Images

#### What is it?

A **Route Handler** (`/api/og/route.tsx`) that uses Next.js's built-in `ImageResponse` from `next/og` to generate a custom `.png` image at request time using JSX and inline styles — no canvas, no Puppeteer.

#### What is it used for?

Generating personalised OG images on the fly based on URL query parameters — for example, a unique card per blog post with its title and description.

#### How `ImageResponse` works

`ImageResponse` accepts a JSX tree and renders it to a PNG image server-side using the **Satori** engine. It supports a subset of CSS (flexbox, text, gradients, opacity) but **not** the full CSS spec.

#### Implementation: the Route Handler

```tsx
// app/api/og/route.tsx
import { ImageResponse } from "next/og";
import type { NextRequest } from "next/server";

// Edge runtime renders faster & cheaper than Node for image generation
export const runtime = "edge";

export async function GET(request: NextRequest) {
  const { searchParams } = new URL(request.url);

  // Read query params with fallback defaults
  const title = searchParams.get("title") || "Kiwadev or Code";
  const description = searchParams.get("description") || "Kiwadev Description";

  return new ImageResponse(
    (
      <div
        style={{
          display: "flex",
          height: "100%",
          width: "100%",
          alignItems: "center",
          justifyContent: "center",
          flexDirection: "column",
          backgroundImage: "linear-gradient(135deg, #667eea 0%, #764ba2 100%)",
          fontSize: 60,
          letterSpacing: -2,
          fontWeight: 700,
          textAlign: "center",
          color: "white",
          padding: "0 120px",
        }}
      >
        <div style={{ marginBottom: 20 }}>{title}</div>
        <div
          style={{
            fontSize: 30,
            fontWeight: 400,
            opacity: 0.8,
            lineHeight: 1.4,
          }}
        >
          {description}
        </div>
      </div>
    ),
    {
      width: 1200,  // standard OG image width
      height: 630,  // standard OG image height
    }
  );
}
```

**Key capabilities of `ImageResponse`:**

| Feature | Detail |
|---------|--------|
| JSX rendering | Write layout as React JSX — familiar syntax |
| Flexbox support | Full flexbox layout model supported |
| Gradients & colors | CSS `backgroundImage` gradients work |
| Dynamic text | Inject any string from query params, DB, or API |
| Custom fonts | Pass `fonts` option to load `.ttf`/`.otf` files |
| Standard size | `1200 × 630` is the recommended OG image dimension |
| Edge runtime | Add `export const runtime = "edge"` for faster cold starts |

#### Implementation: assigning the dynamic OG image in a page

Use `generateMetadata()` (the dynamic counterpart to `metadata`) to build the OG URL with query parameters and attach it to the route:

```typescript
// app/blog/[slug]/page.tsx
import type { Metadata } from "next";

type Props = {
  params: { slug: string };
};

export async function generateMetadata({ params }: Props): Promise<Metadata> {
  const { slug } = params;

  // Build the URL pointing to our /api/og route handler
  const ogImageUrl = `http://localhost:3000/api/og?title=${encodeURIComponent(
    `Blog ${slug}`
  )}&description=${encodeURIComponent("This is a blog page")}`;

  return {
    title: `Blog ${slug}`,
    description: `This is about ${slug}`,
    openGraph: {
      title: `Blog ${slug}`,
      description: "This is a blog page",
      images: [
        {
          url: ogImageUrl,
          width: 1200,   // match ImageResponse width
          height: 630,   // match ImageResponse height
          alt: "OG Image Alt",
        },
      ],
    },
  };
}

export default function BlogPage({ params }: Props) {
  return <main>Blog: {params.slug}</main>;
}
```

**Why `encodeURIComponent`?**  
Query param values that contain spaces, slashes, or special characters must be URL-encoded or the server will receive a malformed URL. `encodeURIComponent` handles this safely.

#### Full request flow

```
    Browser shares URL
        ↓
    Social platform fetches the page HTML
        ↓
    Reads <meta property="og:image" content="http://.../api/og?title=...">
        ↓
    GET /api/og?title=Blog+hello-world&description=This+is+a+blog+page
        ↓
    Edge Runtime executes ImageResponse(JSX)
        ↓
    Returns a 1200×630 PNG with the dynamic title & description rendered
        ↓
    Social platform displays the image in the link preview
```
